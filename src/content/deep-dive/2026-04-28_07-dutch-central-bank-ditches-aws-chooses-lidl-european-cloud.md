# 荷兰央行弃用 AWS，选择 Lidl 旗下欧洲云

> 来源: Techzine | HN 热度: 294 points, 128 comments

## 事件背景

荷兰中央银行（De Nederlandsche Bank, DNB）即将与德国 Schwarz 集团（Lidl 超市母公司）旗下的 IT 部门 Schwarz Digits 签署重大云服务合同。Schwarz Digits 通过其 Stackit 云平台定位为美国超大规模云厂商的欧洲替代方案。这一决定的背景是：DNB 和荷兰金融市场监管局（AFM）去年曾警告荷兰金融行业对外国（尤其是美国）IT 服务提供商过度依赖，而 DNB 本身也承认高度依赖美国服务商。地缘政治紧张加剧了这一担忧——例如，海牙国际刑事法院的检察官曾被特朗普政府切断微软邮箱账户。Stackit 的核心卖点是所有数据受欧洲法律管辖，不受美国《云法案》（CLOUD Act）约束，Schwarz 集团最近还宣布在 Lübbenau 投资110亿欧元建设大型数据中心。

## 核心观点

这一事件标志着欧洲数字主权战略从"讨论"进入"行动"阶段。荷兰央行作为高度监管领域的重要机构，其选择具有强烈的示范效应。然而，务实的挑战不容忽视——DNB 总监 Steven Maijoor 去年10月坦承欧洲云"还没有美国的那么强健或高质量"。Stackit 最初是 Lidl 和 Kaufland 内部的 IT 系统，现在也服务于德意志铁路、SAP 和拜仁慕尼黑等外部客户，但与拥有20年发展积累的 AWS、Azure 和 GCP 相比差距仍然明显。德国石勒苏益格-荷尔斯泰因州从微软向开源环境迁移时的困难已经证明了这种转型的复杂性。核心问题在于：**为了数据主权，欧洲机构愿意承受多大的功能和性能折衷？**

## 社区热议

HN 社区128条评论围绕云依赖和自主权展开了务实的讨论：

1. **早就应该避免锁定** 一位用户回忆道："多年前我就主张，与其把自己绑进 Amazon 生态系统（S3、EC2、DynamoDB 等专有产品），不如直接用虚拟机加开源产品。人们看我的眼神就像看到水在燃烧——但那样做本可以让脱离美国依赖变得容易得多。只需迁移 VM 就行。"

2. **云厂商的锁定策略** 用户详细描述了典型的锁定路径："2016年——用 EC2 吧，只是 VM，随时能搬走。2018年——你在 EC2 上自己跑 PostgreSQL？用我们的托管方案吧。2020年——你已经用了我们18个服务了，为什么不用我们的 IAM？2024年——你已经深度锁定了，来试试 DynamoDB 优化成本吧。此时没有人会质疑下一个销售人员推的工具，因为工程师看到公司没有迁移策略。" 另一位用户补充道最新阶段："2025年——你开始使用带预签名 URL 的 S3 直接服务客户。好了，现在你的客户也被锁在 AWS 上了。"

3. **"PPT 架构师"批判** 多位用户批评了导致云锁定的组织文化。一位用户描述道："我遇到的大多数 CTO 都是我所谓的'画方块架构师'——他们喜欢画方块、把一个方块移到另一个方块里、在两个方块之间画条线，然后宣布困难的部分已经完成。" 另一位更直接："我们叫这些人 PowerPoint 架构师。他们已经很久没写过代码或构建过任何东西了。"

4. **自建的现实可行性** 一位创业者分享了自建基础设施的经验："我在办公室用自建 GPU 服务器运行创业项目，备份到云端。只需付 IP 地址费用，电费包含在房租里……这台机器服务的用户已经比我很多老客户用昂贵的 K8s 集群服务的还多。而且，搭建它的过程很有趣——听着那东西在角落嗡嗡运转就很舒服。"

## 行业影响

荷兰央行的选择对欧洲云计算市场和全球数据主权格局具有标志性意义。**欧洲主权云加速**：作为金融监管机构的"以身作则"，DNB 的选择将鼓励更多欧洲政府和金融机构评估本土云替代方案，Stackit 的客户名单可能快速增长。**美国云厂商的欧洲策略调整**：AWS、Azure 和 GCP 可能加速推出"欧洲主权云"产品线，加强数据本地化和合规承诺。**开源与自主部署复兴**：对云锁定的担忧正在推动企业重新评估基于开源软件和虚拟机的可移植架构。**功能差距的商业机会**：Stackit 与超大规模云厂商之间的功能差距为欧洲 SaaS/PaaS 公司创造了构建"补充层"的市场机会。但同时也要看到，从美国云迁移到欧洲云并非简单的"换供应商"——石勒苏益格-荷尔斯泰因的前车之鉴表明，这是一个需要数年时间、大量投入和组织变革的系统性工程。
