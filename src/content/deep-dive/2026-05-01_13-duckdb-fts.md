# Full-Text Search with DuckDB

- **来源**: Hacker News (front, 109 points)
- **原帖**: https://news.ycombinator.com/item?id=47966284
- **外链**: https://peterdohertys.website/blog-posts/full-text-search-w-duckdb.html

## 事件背景

Peter Doherty 写的 DuckDB 系列第二篇，专门评估 DuckDB 内置的全文搜索（FTS）扩展能力。作者背景是熟悉 Elasticsearch 与 Postgres FTS（含 pgvector / pg_search）的工程师，文章带横向对比视角。HN 上获得百分以上 upvote，反映社区对"轻量级 FTS"的持续兴趣，特别是把 DuckDB 当 OLAP/数据探索引擎的人。

## 核心机制

文章先做了一遍简短的 FTS 原理回顾：stemming、stop-words、accent stripping、Okapi BM25 评分（k₁ 控制词频饱和、b 控制文档长度归一）。随后展示 DuckDB 用法：

```sql
INSTALL fts; LOAD fts;
PRAGMA create_fts_index('articles', 'id', 'title', 'body');
SELECT *, fts_main_articles.match_bm25(id, 'rust async') AS score
FROM articles WHERE score IS NOT NULL ORDER BY score DESC;
```

作者复盘 DuckDB FTS 当下能做什么：基础 BM25 + 简单 tokenize/stem/stop-words；做不到的：高亮匹配位置（缺类似 Postgres `ts_headline`）、phrase query、可插拔同义词词典、向量混合检索。

## 竞品对比

- **Postgres `tsvector` + GIN**：覆盖度更全，支持 ts_headline、ts_rank_cd，加上 pgvector 可做 hybrid。
- **Elasticsearch / Meilisearch / Tantivy / Typesense**：分布式、生产级，但部署成本远超嵌入式场景。
- **SQLite FTS5**：是 DuckDB FTS 的最直接对标，同样轻量，但 BM25 调参更隐晦。
- DuckDB 的优势：和 Parquet/CSV/Lakehouse 数据无缝混合，分析查询里直接拼 FTS 评分。

## 反馈与局限

社区共识：作为"一次性数据探索"或"中小型嵌入式应用文本搜索"很合适，但**不要用作生产级搜索后端**。作者明确指出 highlight 缺失逼他用 tmux + 字符串搜索补救，相当不优雅。HN 评论提到 DuckDB 团队已经 tracking phrase queries 与 vector hybrid 的 issue，预计未来版本会跟进。对量化/数据团队的实用启示：DuckDB FTS 适合做 research notebook 内对论文/邮件的本地全文索引，不必再起 Elasticsearch。
