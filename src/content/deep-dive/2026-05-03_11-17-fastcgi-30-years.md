# FastCGI: 30 years old and still the better protocol for reverse proxies

- HN: 419 points
- 链接: https://www.agwa.name/blog/post/fastcgi_is_the_better_protocol_for_reverse_proxies

## 事件背景
SSL.com 创始人 Andrew Ayer 写了一篇带挑衅意味的工程博客：在 HTTP/2 与 gRPC 大行其道的今天，1996 年的 FastCGI 协议在「反向代理 → 应用服务器」这一垂直场景上仍然是更优解。HN 把它推上首页，引发关于「为什么我们都在用 HTTP-based reverse proxy 转发」的反思。

## 核心观点 / 产品机制
- HTTP 在「客户端 ↔ 反代」这一段是必须的；但「反代 ↔ 应用」一段往往只重传同样的 HTTP 头部，浪费了 parse 成本；
- FastCGI 设计目标恰好是「单 socket 上多路复用请求」，且头部是二进制定长 + 名值对，省去 ASCII parser；
- 安全：FastCGI 用 unix domain socket，应用进程不需要监听公网端口；身份/参数由代理填充，避免 HTTP smuggling；
- 现代 web server (nginx, Caddy) 与各语言 runtime (PHP-FPM, Python uWSGI) 仍内建 FastCGI 支持；
- HTTP/2 引入 multiplexing 后，FastCGI 优势收窄，但二进制 + 短头部 + 不需要 TLS 仍然让它保持效率。

## 社区热议与争议点
- 拥护派（顶赞）：「20 年了 PHP-FPM 仍然是世界上跑得最稳的 web 协议组合」；
- 反对派：「gRPC over Unix socket 一样高效，且工具链更现代」；
- 平衡派：建议根据生态选择——Python/Ruby/PHP 选 FastCGI/uWSGI，新项目选 HTTP/2 keep-alive 也没大错；
- 进阶派：讨论 mTLS + HTTP/3 + Wireguard sidecar 的多层组合是否还需要 FastCGI 这一层抽象。

## 行业影响与未来展望
随着「应用 sidecar / service mesh」成为主流，反代 → 应用之间的协议层正在被重新审视。FastCGI 给的启示是：协议越简单越易于审计，二进制定长帧让 fuzz / 安全检测更可靠。短期内 FastCGI 不会取代 HTTP，但作为「轻量、低延迟、易实现」的备选，它仍是有价值的工程选择。开发者可以在内部 RPC、CI 流水线 worker 调度等场景中重新评估 FastCGI / SCGI。
