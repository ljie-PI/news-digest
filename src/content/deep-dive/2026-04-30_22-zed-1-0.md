# Zed 1.0

**链接**：[https://zed.dev/blog/zed-1-0](https://zed.dev/blog/zed-1-0) · [HN 讨论](https://news.ycombinator.com/item?id=47949027) · 1396 points · 442 comments

## 事件背景

2026 年 4 月 29 日，Zed 团队（Atom 与 Tree-sitter 的原作者团队）正式发布了代码编辑器 Zed 1.0。这是从 Atom 项目"自我重置"以来五年的工程结晶——他们用 Rust 从零开始，抛弃了 Electron 的网页堆栈，转而像构建电子游戏一样组织整个应用：所有 UI 数据被馈送给 GPU 上的着色器（GPUI 框架）。1.0 标志着 Zed 已积累 100 万行代码、覆盖 macOS / Linux / Windows，可与 VS Code 在功能广度上掰手腕。

## 核心机制

Zed 的差异化在于**自有底座**：GPUI（Rust 写的 GPU UI 框架）+ 协同编辑层 + AI-native 设计。1.0 的核心特性包括：Git 集成、调试器、SSH 远程开发、并行运行多个 AI agent、字符级别预测的 Edit Prediction，以及自家提出的 Agent Client Protocol（ACP）——一个让任意 agent 后端接入编辑器的开放标准。Mistral 的 Vibe、Anthropic 的 Claude Code 等已先后接入 ACP。

## 社区热议与争议点

HN 讨论区呈现典型的 Cursor / VS Code / Zed 三方拉锯：

- **Pro**：性能党盛赞 GPUI 的丝滑滚动和大文件处理，认为这是十年来"第一个本地原生且全功能"的现代编辑器。
- **Con**：批评者指 Zed 的 Vim 模式、插件生态仍弱于 VS Code；商业模式（团队协同付费 + AI 功能 token 计费）让人担心 Atom 的命运重演。
- **争议**：是否还需要又一个编辑器？反方认为在 Cursor / Windsurf / Continue 等 AI 编辑器混战的当下，"GPU 渲染"卖点已不够；正方则主张只有 Zed 把 ACP 的协议化抽象做对了，未来 agent 编排会让它脱颖而出。

## 行业影响与未来展望

1.0 在象征意义上重要：它意味着一支团队真正用 Rust + GPU 把"非 Electron 的 IDE"做到生产可用，证伪了"网页栈不可绕过"的行业惯性。短期对 VS Code 形成的替代压力有限，但 ACP 若被 IDE 厂商广泛采纳，将削弱 Cursor 这类基于 fork 模式的玩家的护城河。值得关注 Zed 在企业市场的协作功能是否能转化为可持续收入。
