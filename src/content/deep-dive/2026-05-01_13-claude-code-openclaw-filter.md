# Claude Code 因提交记录中的 “OpenClaw” 字样出现异常扣额/拒答

> Hacker News | 2026-05-01

## 事件背景

HN 当天最热的故事不是新模型，而是一个非常“工程化”的事故：多位用户复现发现，只要 git commit message 或上下文里出现 `OpenClaw` 这类敏感字符串，Claude Code 就可能立刻中断、拒绝继续，甚至把一次几乎没有实际输出的会话记成高额 usage。触发点之所以危险，在于 Claude Code 会把近期 git 历史、项目文件乃至网页内容一并带进提示上下文，因此这不是“你主动输入了某句话”的单点问题，而是任何上游文本都可能把工具链打坏。

## 核心观点

这件事暴露的不是单个过滤规则写得粗糙，而是 agent 产品把“外部不可信文本”与“高优先级系统控制”混在了一条上下文通道里。只要服务端过滤器或路由逻辑对某些字符串有特殊处理，而模型又无法稳定地区分“说明文本”“代码注释”“commit message”“恶意注入”，就会出现低成本的拒绝服务与异常计费。对 coding agent 来说，这比普通聊天里的提示注入更糟，因为它会污染自动审阅、代码补全、批量重试和长链路工具调用。

## 社区热议

HN 评论区最集中的讨论有三类。第一类认为这是典型的 prompt injection / policy routing 设计失误：既然任意仓库文本都可能进入上下文，就不该让某个字符串直接影响会话计费或执行权。第二类担心供应链层面的“隐形投毒”——有人完全可以把类似字样塞进 README、网页甚至白底白字的文档里，专门破坏别人的 agent。第三类讨论则更抽象：为什么新一代 AI 工具一次次重复邮箱、浏览器、安全沙箱这些老系统早就总结过的分层经验。

## 行业影响

这件事对整个 agent 赛道是一个硬提醒：真正的竞争壁垒不是“模型更聪明”，而是“上下文边界更稳”。未来所有 coding agent 都必须更明确地区分系统提示、可信仓库规则、用户输入和外部抓取内容，最好在产品层加入 provenance 标注、隔离执行和异常计费回滚。不然随着 agent 越来越多地读取仓库、网页、issue 和聊天记录，类似的字符串触发事故只会越来越频繁。
