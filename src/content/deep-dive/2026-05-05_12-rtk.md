# rtk-ai/rtk — 让 LLM 少吃 60-90% Token 的 CLI 代理

- 项目链接: https://github.com/rtk-ai/rtk
- 官网: https://www.rtk-ai.app
- 时间段新增 stars: +4,504 (this week) | 语言: Rust

## 1. 定位与痛点剖析

rtk 是一个**高性能 CLI 代理**，目标是把 coding agent（Claude Code、Cursor、Codex、Gemini CLI、Cline、Windsurf、Antigravity 等）每次执行 shell 命令时返回到上下文窗口里的"垃圾输出"压缩 60-90%。痛点非常明确：今天的 agent 跑一次 `git status`、`pytest`、`docker ps`、`kubectl logs` 动辄几千 token；一个中等会话里 30 分钟可烧掉整个 200K 窗口。rtk 在 agent 执行 Bash 之前透明改写命令（`git status` → `rtk git status`），返回经压缩后的等价信息——agent 只看到压缩结果，模型完全感知不到中间层。

## 2. 核心架构与技术细节

单 Rust 二进制，零运行时依赖（无需 Node/Python），<10ms 开销。支持 100+ 命令包括 `ls`/`tree`/`cat`/`grep`/`rg`/`git`/`gh`/`docker`/`kubectl`/`cargo`/`pytest`/`npm test`/`tsc`/`ESLint`/`AWS CLI` 等。压缩策略分四种：
- **Smart Filtering**：删注释、空白、模板话；
- **Grouping**：把同类按目录/规则聚合；
- **Truncation**：保留有效上下文，砍冗余；
- **Deduplication**：折叠重复日志带计数。

实测数据（README 自报）：`ls` 减 80%，`cat` 减 70%，`pytest` 减 90%，`git add` 仅返回 `"ok"` 二字。集成方式是注入 agent 的 hook —— `rtk init -g --gemini`、`--codex`、`--agent cursor` 等命令一键改写 `settings.json` 的 PreToolUse hook，agent 重启后自动生效。

## 3. 竞品对比与生态站位

正面对手是 **mksglu/context-mode**（同周 trending #9，做的是 MCP 端的 context 管理 + 沙箱）。两者哲学不同：context-mode 通过 sandbox + SQLite event store 改造 agent 自己的工作模式，rtk 则在 OS 层面做"输出压缩"，对 agent 几乎无侵入。还有一类对手是 IDE 自带的 ripgrep/rg 集成（Cursor/Windsurf 内置），但那些只覆盖搜索单一场景。rtk 的优势在于 **覆盖面广 + 一行 init**，劣势在于只对"通过 Bash tool 调用"的命令生效——Claude Code 自带 Read/Grep/Glob 的内部工具调用绕过 rtk。

## 4. 开发者反馈与局限性

正面：社区普遍认可"立即省钱"的体验；Brew/cargo 一键装；Discord 活跃。
争议与局限：
- **绕过路径多**：agent 直接调用 IDE 内置工具不会被 hook；
- **Windows 体验差**：原生 Windows 下 hook 有限，官方建议走 WSL；
- **黑盒压缩可能丢信息**：尤其在 debug 罕见错误时，"截断"可能让 agent 看不到关键栈帧；
- **维护成本**：100+ 命令的输出格式经常变（如 `gh`、`kubectl` 升级），适配压力大。
