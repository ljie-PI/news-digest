## 2) 核心观点与机制

Raymond Chen 在原文中追溯了这段历史的全貌：

**1973年 → CP/M 时代**：CP/M 作为早期微机主流操作系统，根本没有环境变量的概念。程序配置完全依赖二进制补丁（patching）——用户直接修改可执行文件的机器码字节来适配硬件（如打印机）或更改行为。WordStar 甚至随手册附带"哪些字节该改成什么值"的说明。

**1981年 → MS-DOS 诞生**：MS-DOS 的设计深受 CP/M 启发，甚至核心目标就是让 CP/M 程序能机器翻译到 8086 平台。MS-DOS 新增了环境变量机制，但早期程序都是从 CP/M 移植而来，根本没人用环境变量。

**后来**：随着原生 MS-DOS 程序增多，开发者发现环境变量可以用来存储配置。在"市场混乱"中，`TEMP` 和 `TMP` 同时冒了出来，成了两大主流选择——**没有统一标准**。

**关键分水岭**：
- MS-DOS 2.0 引入管道（pipe）机制，需要一个临时文件目录。COMMAND.COM 的开发团队选择了 `TEMP`。
- 但 Windows 的 `GetTempFileName` 函数的原作者选择了优先检查 `TMP`，其次才是 `TEMP`。

结果就是：**不同程序遵循不同的优先级规则**。Windows 原生程序通常走 `GetTempFileName`，所以偏向 `TMP`；而一些老派 DOS 程序则优先 `TEMP`。即使到今天，Windows 的环境变量对话框里两者依然并存，就像"Adidas vs Puma 的极客版本"，谁也没法干掉谁。

## 4) 行业影响与未来展望

这个看似微不足道的命名分歧，实际上是**技术债务（technical debt）**的经典教科书案例：

1. **路径依赖的不可逆性**：一旦两个变量在生态系统中生根，任何"统一"的尝试都会破坏无数遗留程序。Windows 作为平台，其核心竞争力之一就是向后兼容，这使得"纠正历史错误"的代价极高。

2. **跨平台开发的隐性成本**：TMP/TEMP 的分歧只是冰山一角。评论区有人类比指出，Unix 世界也存在 `http_proxy` vs `HTTP_PROXY` 的不一致，Linux 移植到 Windows 的程序仍然沿用 `TMP` 或 `TEMP`。环境变量的命名标准化问题在跨平台时代依然悬而未决。

3. **现代配置的启示**：XDG Base Directory Specification（2003 年）试图在 Linux 世界统一配置、缓存、数据目录的路径规范，但至今仍有大量程序（包括 Firefox 这样的巨头）只是将家目录污染从 `~/.appname` 搬到了 `~/.config/appname`，并未真正分离缓存、日志和配置。TMP/TEMP 的故事提醒我们：**标准只有在被广泛采纳时才有意义**，而采纳的动力往往来自平台方的强制力（如 macOS 的沙盒规则），而非社区自愿遵守。

展望未来，AI 生成代码的普及或许能缓解这类问题——评论区有人半开玩笑地说："不管怎么说，AI 比人类更可能遵守这些标准。" 但归根结底， legacy 系统的惯性意味着 TMP 和 TEMP 很可能将伴随 Windows 走到尽头。
