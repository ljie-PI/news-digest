# Red Squares：把 GitHub 宕机画成"贡献图"的讽刺项目

## 项目简介

**Red Squares** 是一个讽刺性数据可视化项目，将 GitHub 官方状态页记录的宕机事件映射到类似 GitHub 贡献热力图的界面中——只不过绿色方块（代表代码贡献）被替换为**红色方块**（代表平台宕机）。项目通过颜色深浅表示宕机严重程度，直观展示了 GitHub 作为全球最大的代码托管平台在过去一段时间内的稳定性表现。

## 关键数据发现

### 工作日 vs. 周末宕机模式

> "GitHub 工作日正常运行率 **89.3%**，周末 **96.5%**。事件波及 **62% 的工作日** 和 **11% 的周末"。—— [ahstilde](https://www.aakash.io/tech-chase/github-and-claude-are-down-three-out-of-four-days)

> "周二到周四是危险区。周日简直是另一个服务。"

- **最糟糕的一天**：2025 年 11 月 20 日（周四），报告宕机时长 1.1 天
- **注意**：由于多服务聚合统计，24 小时周期内可能出现超过 24 小时的"可能宕机时间"

### Claude 呈现相同模式

- **工作日 92.5% / 周末 97.8%** 的正常运行率——暗示问题可能不仅限于 GitHub，而是更广泛的 AI 基础设施负载模式

## 核心争论：宕机原因何在？

### 观点一：负载/扩容问题

> "比较一下公共 GitHub 与 GitHub Enterprise Cloud 的状态页。企业版数据好得多……如果问题不围绕负载，我预期会在企业版上看到同样的正常运行率问题。" —— **u_fucking_dork**

> "按他们自己的说法，正朝着 **30 倍年度交易量** 前进" —— **baq**

> "Agent 正在全球范围内更快地提交代码……从每年 10 亿次 commit 到预计 **140 亿次以上**，仅用一年" —— **fareesh**

### 观点二：管理/技术债/系统性失败

> "在连网站都保持不了正常运行的时候，重写 UI 和强推 AI 工具是浪费时间……没人针对工程师，我们抨击的是这样一个事实：由于网络效应而成为事实垄断的系统，正在把新功能或取悦所有者置于核心服务之上。" —— **dijit**

> "他们的技术栈确实烂，而且多年来他们一直在公开辩护，相当傲慢……Azure 确实糟糕，而且被强加给团队" —— **dijit**

> "我期望这家 3 万亿美元的公司能处理负载，或者至少在顶部放一个醒目的'仅限爱好使用'警告。" —— **senko**

## 关键技术背景

### 部署频率（内部人士透露）

> "GitHub 在工作日期间，部署频率接近 **每 60-90 分钟一次**（部署列车/合并队列批次发出的频率），**24 小时不间断**" —— **rufo**（前员工）

> "过去的情况是，如果没人部署，GitHub 实际上**更不可靠**……存在各种**资源泄漏**，在人们全天部署时看不到，因为那时应用不会不断重启。在 GitHub 假期期间宕机后，我们甚至安排了志愿者在假期期间每天部署一次 GitHub"

### GitHub Enterprise Cloud vs. 公共基础设施

- **标准 GitHub Enterprise Cloud**：与公共 GitHub（github.com）运行在同一基础设施上
- **GitHub Enterprise Cloud with Data Residency**：独立基础设施，通过 `*.ghe.com` 域名访问——**正常运行率更好，但仍基于 Azure**

> "如果你使用 GitHub Enterprise Cloud with data residency，那么你就在独立基础设施上……它的正常运行率肯定比公共云好，但也并非没有问题。" —— **voxic11**

> "它仍然在 Azure 上，所以受制于 Azure 的底层问题" —— **everfrustrated**

## 状态页可靠性之争

### 官方 vs. 独立追踪

> "官方与第三方状态页之间的对比巨大。如果他们的 SLA 条款与现实使用情况如此不同，这些条款怎么合法？" —— **__natty__**

> "他们的服务条款合法，因为条款要求**你，客户**，来追踪他们的可用性是否符合约定的 SLA，并在他们违反 SLA 时追讨赔偿。" —— **xyzzy_plugh**

> "我们与 GitHub 的账户代表和工程人员谈过——**他们没有监控来追踪是否履行了合同 SLA**。他们期望我们记录故障" —— **everfrustrated**

### 聚合方法论问题

> "将'某些 Copilot 聊天模型故障'和'Teams 通知应用宕机'视为与 Git 操作或 Actions 失败同等严重的主要宕机……聚合宕机时间，使得 24 小时周期内可能出现超过 1 天的可能宕机时间" —— **john_strinlai**

> "如果 3 个服务在同一 1pm-2pm 时间段宕机，这被计为 3 小时宕机，尽管开发者只受影响了 1 小时" —— **john_strinlai**

## 结论

Red Squares 表面是一个幽默的数据可视化，实则揭示了**现代数字基础设施的深层张力**：当平台成为开发者经济的"关键基础设施"时，其稳定性责任与商业扩张冲动之间的冲突日益尖锐。GitHub 的宕机模式——工作日高发、与负载强相关、官方状态页与独立监控存在显著差异——指向一个系统性问题：在 AI Agent 爆发式增长的时代，代码托管平台的扩容速度可能正在落后于需求曲线的陡峭化。

对于依赖 GitHub 的企业和开发者而言，这个项目的真正警示在于：**不要把平台的可用性视为理所当然**——尤其是当平台的激励结构更偏向功能扩张而非核心可靠性时。
