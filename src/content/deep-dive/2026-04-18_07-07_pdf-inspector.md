# pdf-inspector

> Firecrawl 出品的高速 Rust PDF 分类与文本提取库——无需 OCR 即可检测 PDF 类型、提取文本、转换 Markdown。

## 定位与痛点剖析

在大规模 PDF 处理管线中，一个关键问题是：**大约 54% 的 PDF 本身就是文本的，但传统做法是把所有 PDF 都送去 OCR 服务处理**，浪费了大量时间和费用。OCR/ML 引擎（docling、marker、mineru）处理 200 个 PDF 需要 2-180 分钟，而这些 PDF 中超过一半根本不需要 OCR。

pdf-inspector 的策略是：**先用毫秒级检测判断 PDF 类型，文本类直接本地提取（~150ms），只把扫描件/图片 PDF 送去 OCR**。这在成本和延迟上带来了数量级的改善。

## 核心架构与技术细节

**处理管线：**
```
PDF bytes → detector → PdfType (TextBased/Scanned/ImageBased/Mixed)
         → extractor → fonts + content_stream + xobjects + links + layout
                     → tables (rectangle-based + heuristic)
                     → markdown (analysis → preprocess → convert → classify → postprocess)
```

**核心能力：**
- **智能分类（~10-50ms）：** 采样内容流检测 TextBased/Scanned/ImageBased/Mixed，返回置信度分数(0.0-1.0) 和逐页 OCR 路由
- **文本提取：** 位置感知提取（字体信息、X/Y 坐标）、自动多栏阅读顺序
- **Markdown 转换：** H1-H4 标题（字体大小比率）、列表（bullet/numbered/letter）、代码块（等宽字体检测）、表格（矩形+启发式双模式）、粗体/斜体、URL 链接
- **表格检测：** 矩形检测（PDF 绘图操作 + union-find）+ 启发式检测（文本对齐），处理金融表格、脚注、跨页续表
- **CID 字体：** ToUnicode CMap 解码，UTF-16BE/UTF-8/Latin-1 编码
- **编码问题检测：** 自动标记损坏字体编码，让调用方回退 OCR

**检测策略：**
- EarlyExit（默认）：扫描所有页面，遇到首个非文本页即停止
- Full：扫描全部页面，精确 Mixed vs Scanned 分类
- Sample(n)：均匀采样 n 页（超大 PDF）
- Pages(vec)：仅检查指定页码

**多语言绑定：** Python（PyO3 + maturin）、Node.js/Bun（napi-rs）、Rust 原生。CLI 工具 `pdf2md` 和 `detect-pdf`。

**性能基准（opendataloader-bench，200 PDFs）：**

| 引擎 | 综合 | 阅读顺序 | 表格 | 标题 | 速度 |
|------|------|---------|------|------|------|
| pdf-inspector | 0.78 | 0.87 | 0.59 | 0.57 | **4s** |
| opendataloader | 0.84 | 0.91 | 0.49 | 0.74 | 11s |
| pymupdf4llm | 0.73 | 0.89 | 0.40 | 0.41 | 18s |
| markitdown | 0.58 | 0.88 | 0.00 | 0.00 | 8s |

**pdf-inspector 是所有引擎中最快的**，200 个 PDF 仅 4 秒。阅读顺序和表格检测优于同类直接文本提取引擎。

**技术特点：** 纯 Rust，无 ML 模型，无外部服务。唯一依赖是 lopdf 做 PDF 解析。文档单次加载共享于检测和提取阶段，避免冗余 I/O。

## 竞品对比与生态站位

| 维度 | pdf-inspector | pymupdf4llm | markitdown | docling/marker (OCR) |
|------|--------------|-------------|------------|---------------------|
| 速度 | 4s/200 PDFs | 18s | 8s | 2-180 min |
| OCR 需求 | 无 | 无 | 无 | 需要 GPU |
| 表格检测 | 0.59 (双模式) | 0.40 | 0.00 | 0.70+ |
| 标题检测 | 0.57 | 0.41 | 0.00 | 0.80+ |
| 扫描件支持 | ❌ 仅路由 | ❌ | ❌ | ✅ |

pdf-inspector 不是要替代 OCR 引擎，而是作为**智能路由前端**——快速筛选出不需要 OCR 的 PDF，为管线节省成本和时间。Firecrawl 的 Fire-PDF 产品将 pdf-inspector 与神经布局模型和 OCR 结合，处理所有类型的 PDF。

## 开发者反馈与局限性

- 标题检测落后于 opendataloader——许多 PDF 用正文字号的粗体做标题
- 表格检测落后于 OCR 引擎（看不到视觉表格结构）
- 不处理扫描件——仅做分类和路由
