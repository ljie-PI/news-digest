# Claude Code 因提交记录提及 "OpenClaw" 拒绝服务或额外收费

## 事件背景

2026年4月30日，开发者 Theo (@theo) 在 X 上爆料了一个令人震惊的发现：**如果用户的 git 仓库中存在一条提及 "OpenClaw" 的提交记录（commit），Anthropic 的 Claude Code 将要么直接拒绝服务，要么向用户额外收取费用。** Theo 的原话是："This is an empty repo, I'm just calling Claude Code directly. Insanity."

这条推文迅速在开发者社区引发轩然大波，8小时内便在 Hacker News 上收获了 856 个赞和 491 条评论。多位用户独立复现了这一现象，而 Anthropic 至今未对此事做出官方解释。

## 核心观点：关键词触发机制与计费陷阱

问题的核心在于 Claude Code 的一个隐藏检测机制：当系统在用户的项目上下文（包括 git 提交记录、代码文件或文档）中扫描到 `openclaw` 字样时，会触发某种预设响应。根据用户实测，结果可能是：

1. **服务拒绝**：立即断开连接，并将用户会话使用率标记为 100%
2. **强制计费**：抛出 "You're out of extra usage" 错误，暗示用户需要购买额外额度才能继续使用
3. **隐性扣费**：用户 `flutas` 发现，即便服务看似正常运行，单次包含该关键词的提示词也悄悄消耗了 $0.20 的"额外使用量"

值得注意的是，该机制的触发方式极为粗糙——多位用户指出，它似乎不区分上下文：只要在 prompt 的任意位置出现目标字符串，无论是有意为之还是无意撞见，都会触发。

## 社区热议与争议点

### 争议一：可复现性存疑 —— 是故意为之还是工程事故？

**支持方（可复现）**：用户 `abdullin` 提供了完整的复现步骤，在一个空仓库中仅通过一条包含 `{"schema": "openclaw.inbound_meta.v1"}` 的 commit，就触发了"Immediate disconnect and session usage went to 100%"。`margalabargala` 则收到了更明确的计费提示："You're out of extra usage. Add more at claude.ai/settings/usage and keep going."

**反对方（无法复现）**：然而 `isoprophlex`、`deaux` 和 `gkbrk`（使用 Opus 4.7 模型）均表示无法复现。`novaleaf` 分析认为这可能是 Anthropic 计费系统中"额外使用量分段逻辑"的 bug，暗示它可能是一个 A/B 测试或正在回滚的临时措施。

这一争议的本质是：Anthropic 是否故意在代码中埋下了针对竞争对手 OpenClaw 的"地雷"？从现有证据看，即使是，其工程实现也极其业余。

### 争议二：拒绝服务攻击（DoS）的可能性

这是讨论最激烈的议题。用户 `petercooper` 最先提出了一个大胆设想："反 AI 的项目维护者是否可以在文档或提交记录中偷偷植入这类标识符，从而破坏使用 Claude Code 的开发者？"

**正方（激进立场）**：`SlinkyOnStairs` 的回复堪称精辟："没有'如果'。他们完全可以做到。prompt 的各部分之间没有隔离，你把那段文字藏进去，放在哪里都能生效。"他甚至指出，"你甚至不需要隐藏它，大大方方放在明处就是一次拒绝服务攻击，还可以自动化实施。" `giancarlostoro` 进一步发挥："你甚至不需要把代码放进项目里，在你的博客文章里加一段白色字体的隐藏文字，Claude 如果在研究任务中读了你的网站，就等于报废了别人的 Claude 会话。"

**反方（合规与道德立场）**：`bluefirebrand` 反驳道："如果项目明确拒绝 AI 协助的 PR，你却还要用 AI，那某种程度上是你自找的。" `amarant` 则从开源精神角度发声："即便你不想要 AI 辅助的 PR，破坏想 fork 你项目的所有人也不太符合开源精神。" 而 `waych` 则从法律角度提醒："在现实世界中，留下可能伤害无辜者的陷阱是一种法律责任，甚至本身就可能构成犯罪。"

### 争议三：用户自由 vs 厂商控制 —— 这算不算 DRM？

**反 Anthropic 方**：`bko` 将这一机制定性为"与 DRM 无异、对用户充满敌意"的行为，质问"我们是不是已经放弃了'你对自己购买的软件拥有自由使用权'这一理念？"`oarsinsync` 补充道："如果我 clone 了一个开源仓库，我理应可以自由地使用 LLM 审查它。故意破坏 LLM 使用的项目，已经不再符合'自由软件'的定义。"

**支持 Anthropic 方**：`joemi` 则提出了另一种视角："你依然可以自由地使用仓库本身。限制发生在 LLM 端，所以归根结底是 LLM 的'错'。如果你不想遇到这种限制，那就换一个没有这种限制的 LLM。"

这场辩论已经超越了单一事件，触及了一个更深层的问题：**AI 工具提供商是否有权通过技术手段干预用户对第三方开源代码的使用方式？**

## 行业影响与未来展望

### 1. 信任危机

`chillfox` 一针见血地指出："AI 是科技行业一个新的产品类别，而每一个新的科技产品类别——毫无例外——都拒绝从历史中吸取教训，于是同样的蠢事反复上演。" 这次事件暴露了 Anthropic 产品决策中的重大透明度缺失：用户付费购买的 AI 助手，为何会在没有明确告知的情况下，因为代码中出现了某个特定字符串而拒绝工作或额外收费？

### 2. 技术层面的安全隐患

如果 Claude Code 确实存在这种"关键词触发拒绝服务"的机制，它本质上构成了一个可被第三方利用的 DoS 攻击面。开源项目维护者、竞争对手或简单的恶作剧者都可以通过在 README、commit message 甚至 issue 中植入特定字符串来"封锁"Claude 用户。`frizlab` 已经在自己的项目中实践了这种"防御性"标记，而 `kristopolous` 透露这种 prompt 注入策略在 API 文档中已经悄然出现。

### 3. 竞争与监管

社区普遍认为这一机制与 Anthropic 和 OpenClaw 之间的商业竞争有关。`_blk` 提到 OpenClaw 的使用曾"在最初的封禁后被重新允许"，暗示两家公司之间存在复杂的历史纠葛。如果属实，这种利用技术武器打击竞争对手的做法，在 AI 监管日趋严格的背景下，可能会招致反垄断或消费者保护机构的审查。

## 结论

无论这一机制是 Anthropic 有意为之的"商业防御"，还是一次仓促上线的工程事故，它都揭示了一个令人不安的现实：我们付费购买的 AI 工具，可能在我们不知情的情况下，携带了对竞争对手产品的"免疫排斥"代码。对于开发者和企业用户而言，这一事件是一个严峻的提醒——**当 AI 工具成为工作流的核心环节时，其内部的黑箱逻辑和潜在偏见，可能比我们想象的更深远。**
