## 1. 定位与痛点剖析

### 1.1 核心定位

GitNexus 自称为 **"The Zero-Server Code Intelligence Engine"**（零服务器代码智能引擎）。它的核心使命是将任意代码库索引为一个完整的**知识图谱**——覆盖每一个依赖关系、调用链、功能集群和执行流程，然后通过智能工具（MCP）将这些结构化上下文暴露给 AI Agent，让 AI 在编码时"不再错过任何代码"。

项目用一个精妙的类比阐明了自己的差异化：

> **"Like DeepWiki, but deeper."**  
> DeepWiki 帮助你*理解*代码。GitNexus 让你*分析*代码——因为知识图谱追踪的是每一条关系，而不仅仅是描述。

### 1.2 解决的痛点

当前 AI 编码工具（Cursor、Claude Code、Codex、Windsurf 等）虽然强大，但本质上对代码库结构是"盲人摸象"。GitNexus 诊断了以下典型问题：

- **修改盲区**：AI 修改了 `UserService.validate()`，却不知道有 47 个函数依赖它的返回类型
- **破坏连锁反应**：未经分析就提交的修改导致调用链断裂
- **上下文碎片化**：传统 Graph RAG 给 LLM 原始图边，希望它自行探索；GitNexus 则在索引时就**预计算结构**——聚类、追踪、评分
- **模型能力不等**：小模型因上下文不足而表现差；GitNexus 用工具封装"重活"，让小模型也能获得完整的架构清晰度

### 1.3 两种使用形态

| 维度 | CLI + MCP（推荐） | Web UI |
|------|------------------|--------|
| 用途 | 日常开发（Cursor/Claude/Codex/Windsurf） | 快速探索、演示、一次性分析 |
| 规模 | 任意大小仓库 | 浏览器内存限制（~5k 文件）或后端模式无限 |
| 安装 | `npm install -g gitnexus` | 免安装：gitnexus.vercel.app |
| 存储 | LadybugDB Native（持久化） | LadybugDB WASM（会话级内存） |
| 隐私 | 完全本地，无网络 | 完全浏览器内，无服务器 |

Bridge 模式（`gitnexus serve`）连接两者：Web UI 自动检测本地服务器，无需重新上传或索引即可浏览 CLI 索引的所有仓库。

## 2. 核心架构与技术细节

### 2.1 Monorepo 结构

```
gitnexus/         → npm 包：CLI、MCP Server、HTTP API、索引管道、LadybugDB 图、嵌入
gitnexus-web/     → Vite + React 瘦客户端：图探索器 + AI 聊天
gitnexus-shared/  → 共享 TypeScript 类型和常量
eval/             → 工具使用基准测试框架
.github/          → CI 工作流 + 复合 Action
```

### 2.2 12 阶段索引管道（DAG）

GitNexus 的分析管道是一个**有向无环图（DAG）**，由 12 个明确依赖关系的阶段组成，最终输出 `KnowledgeGraph`：

```
scan → structure → [markdown, cobol] → parse → [routes, tools, orm] → crossFile → mro → communities → processes
```

| 阶段 | 输出 |
|------|------|
| scan | 文件路径 + 大小 |
| structure | 文件/文件夹节点，CONTAINS 边 |
| markdown | Markdown 章节节点，交叉链接边 |
| parse | 符号节点，IMPORTS/CALLS/EXTENDS 边 |
| routes | Route 节点 + HANDLES_ROUTE 边 |
| tools | Tool 节点 + HANDLES_TOOL 边 |
| orm | QUERIES 边（Prisma、Supabase） |
| crossFile | 跨文件类型传播（拓扑导入顺序） |
| mro | METHOD_OVERRIDES + METHOD_IMPLEMENTS 边 |
| communities | Community 节点 + MEMBER_OF 边（Leiden 算法） |
| processes | Process 节点 + STEP_IN_PROCESS 边 |

**DAG Runner** 使用 Kahn 拓扑排序进行编译时类型安全的阶段调度，每个阶段只接收声明的依赖输出，防止隐藏耦合。

### 2.3 双路径调用解析架构

GitNexus 正在经历一场**架构升级**：

- **Legacy Call-Resolution DAG**（6 阶段管道）：`extract-call → classify-form → infer-receiver → select-dispatch → resolve-target → emit-edge`
- **Scope-Resolution Pipeline（RFC #909 Ring 3）**：语言无关的 registry-primary 解析器，已迁移 Python 和 C#

两者通过 **Same-Graph Guarantee** 共存：节点 ID、边词汇表、置信度层级完全一致，下游消费者（MCP 工具、HTTP API、嵌入）无感知。CI 对每条迁移语言的 fixture corpus 运行双路径校验，差异即失败。

### 2.4 四层语言抽象

```
Unified Graph Schema（44 节点类型，21 关系类型）
        ↑
Unified Resolution（3 层名称查找 + MRO 遍历）
        ↑
Language Providers（导入语义、类型配置、导出检查、MRO 策略）
        ↑
Tree-Sitter Queries（每语言 S-expression，统一捕获标签）
```

16 种语言共享同一图模式。每种语言只需实现 `LanguageProvider` 接口，缺失语言在编译期即报错（TypeScript `satisfies Record<SupportedLanguages, LanguageProvider>`）。

### 2.5 技术栈全景

| 层级 | CLI | Web |
|------|-----|-----|
| 运行时 | Node.js（原生） | 浏览器（WASM） |
| 解析 | Tree-sitter 原生绑定 | Tree-sitter WASM |
| 数据库 | LadybugDB Native | LadybugDB WASM |
| 嵌入 | HuggingFace transformers.js（GPU/CPU） | transformers.js（WebGPU/WASM） |
| 搜索 | BM25 + 语义 + RRF | BM25 + 语义 + RRF |
| Agent 接口 | MCP（stdio） | LangChain ReAct Agent |
| 可视化 | — | Sigma.js + Graphology（WebGL） |
| 前端 | — | React 18, TypeScript, Vite, Tailwind v4 |
| 聚类 | Graphology | Graphology |
| 并发 | Worker threads + async | Web Workers + Comlink |

**关键依赖**：
- **LadybugDB**（原 KuzuDB）：嵌入式图数据库，支持向量存储。图模式定义 28 种节点表 + 单一 `CodeRelation` 关系表
- **Tree-sitter**：AST 解析，每语言独立的 S-expression 查询
- **Snowflake arctic-embed-xs**：384 维嵌入模型，基于 SHA1 内容哈希的增量更新
- **Sigma.js + Graphology**：WebGL 图渲染与图数据结构
- **transformers.js**：浏览器端 ML 推理
- **MCP（Model Context Protocol）**：AI Agent 工具接口标准

### 2.6 MCP 工具矩阵（16 个工具）

GitNexus 通过 MCP 向 AI Agent 暴露 11 个 per-repo 工具 + 5 个 group 工具：

| 工具 | 功能 |
|------|------|
| `query` | 进程分组混合搜索（BM25 + 语义 + RRF） |
| `context` | 360° 符号视图——分类引用、进程参与 |
| `impact` | 爆炸半径分析，深度分组 + 置信度 |
| `detect_changes` | Git diff 影响映射 |
| `rename` | 多文件协调重命名（图 + 文本搜索） |
| `cypher` | 原始 Cypher 图查询 |
| `api_impact` | API 路由变更前影响报告 |
| `route_map` | API 路由 → 处理器 → 消费者映射 |
| `tool_map` | MCP/RPC 工具定义和处理器映射 |
| `shape_check` | 响应形状与消费者属性访问不匹配检测 |
| `group_sync` | 跨仓库/服务提取合约并匹配 |

此外提供 **Resources**（`gitnexus://repos`, `gitnexus://repo/{name}/context` 等）和 **2 个 MCP Prompts**（`detect_impact`, `generate_map`），以及 **4 个 Agent Skills**（Exploring, Debugging, Impact Analysis, Refactoring）。

### 2.7 存储模型

```
<repo>/.gitnexus/
  ├── lbug/           # LadybugDB 数据库
  ├── lbug.wal        # 预写日志
  ├── lbug.lock       # 单写入锁
  └── meta.json       # lastCommit, indexedAt, stats

~/.gitnexus/
  └── registry.json   # 全局仓库注册表（MCP 发现）
```

Multi-Repo 架构使用全局注册表：一个 MCP Server 可服务多个已索引仓库，无需 per-project 配置。LadybugDB 连接惰性打开，5 分钟不活动后驱逐（最多 5 个并发）。

### 3.2 差异化定位

**GitNexus 的核心差异化在于"预计算关系智能"（Precomputed Relational Intelligence）**：

- **传统 Graph RAG**：给 LLM 原始图边，指望它自己探索 → 需要多次查询链，token 消耗大，小模型容易遗漏
- **GitNexus**：在索引阶段就完成聚类、追踪、评分 → 工具一次调用返回完整上下文 → LLM 不会遗漏，token 效率高，小模型也能胜任

这意味着 GitNexus 不是又一个代码搜索工具，而是**给 AI Agent 装上了一个"外接大脑"**——一个结构化的、可查询的、关系完整的代码知识图谱。

### 3.3 生态站位

GitNexus 选择了一个精妙的生态位：

1. **不是编辑器**：它不替代 Cursor 或 VSCode，而是*增强*它们
2. **不是模型**：它不训练 LLM，而是给 LLM 提供结构化上下文
3. **不是托管服务**：零服务器架构意味着完全的数据主权
4. **是基础设施层**：在代码库和 AI Agent 之间建立一座"关系桥梁"

当前已支持 5 大编辑器：Claude Code（完整支持：MCP + Skills + Pre/Post ToolUse Hooks）、Cursor、Codex、Windsurf、OpenCode。社区还衍生出 pi-gitnexus 插件和稳定运维工作流。

### 4.3 已知技术局限性

从 ARCHITECTURE.md 和 README 中可识别的局限：

1. **重载方法解析**：使用 arity 后缀（`#<paramCount>`）区分，同 arity 时用 type-hash。无类型语言（Python、Ruby、JS）仅用 arity，精度有限。添加重载会导致节点 ID 变化，影响稳定性
2. **增量索引尚未实现**：Roadmap 中标记为"Actively Building"，当前每次 `analyze` 需全量重建（除非 `--force` 检测到 HEAD 未变时跳过）
3. **Web UI 浏览器内存限制**：纯前端模式约 5k 文件上限，需 `gitnexus serve` 后端模式突破
4. **嵌入生成可选且有限制**：>50k 节点时默认跳过嵌入，影响语义搜索质量
5. **AST Decorator 检测**：仍在开发中（Spring Boot @Controller、@Get 等框架模式）
6. **部分语言支持不完整**：Swift 无 import/named binding 支持，C/C++ 的 import 解析缺失
7. **PolyForm Noncommercial 许可证**：限制商业使用，企业版需联系 akonlabs.com

### 4.4 企业版与商业化

GitNexus 采用**开源核心 + 企业增值**模式：
- **开源版**：PolyForm Noncommercial License，个人和非商业使用
- **企业版**（akonlabs.com）：SaaS 或自托管，包含 PR Review 自动爆炸半径分析、自动更新 Code Wiki、自动重新索引、多仓库统一图谱、OCaml 支持、优先功能/语言支持
- **Docker 镜像**：双重签名（Cosign keyless），GHCR + Docker Hub 镜像一致，供应链安全投入显著
