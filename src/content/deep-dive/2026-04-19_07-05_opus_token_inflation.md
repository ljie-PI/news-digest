# Opus 4.7 vs 4.6 Token 通胀约 45%：用户感受到的真实成本飙升

**367 points | 385 comments**

## 事件背景

开发者 Bill Chambers 创建了一个社区驱动的 Token 成本计算器网站（tokens.billchambers.me），收集匿名用户提交的真实提示词，比较 Opus 4.6 和 4.7 的 token 消耗差异。社区数据表明 4.7 相比 4.6 的 token 膨胀率约为 45%，与第一条新闻中的独立测量结果高度吻合。这直接触发了 Claude Code 用户群体的广泛不满。

## 核心观点/产品机制

该网站的核心机制是众包验证：用户提交真实的 prompt，系统自动计算两个模型的 token 数量并展示对比。网站开源（github.com/bllchmbrs/tokensmatter），存储行仅包含匿名提交 ID，不与 Anthropic 有任何关联。

但 token 数量只是成本等式的一部分。一位用户通过 Artificial Analysis 数据指出了更完整的图景：4.7 产生显著更少的输出 token，推理成本也大幅降低——推理成本从 4.6 到 4.7 几乎减半。综合来看，4.7 的总成本与 4.6 大致持平，甚至略低。但 4.5 的总成本只有约一半。

## 社区热议与争议点

**1. 强制自适应思考导致质量下降**：最激烈的批评来自对 4.7 强制启用 adaptive thinking 的不满。一位用户贴出了与 Claude 的对话记录，展示模型在"最大 effort"设置下仍然"敷衍了事"——模型自己承认"我在模式匹配，没有真正阅读你的代码"。该用户指出这与两周前 bcherny 建议禁用 adaptive thinking 的情况如出一辙："即使 API 用户也无法禁用它。Opus 在启用 adaptive thinking 时完全不可信。"

**2. 速率限制消耗急剧加速**：一位 Max 5x 计划用户报告："我在 2 小时内就耗尽了 5 小时的限额。过去每次交互消耗约 1-2%，现在达到 5%。"另一位尝试批量模式的用户发现"5 分钟内消耗了 30% 的限额"。

**3. 缓存机制的隐性成本**：有用户发现如果提示之间等待超过 5 分钟不进行压缩或清理，会为重新初始化缓存付费。更令人困惑的是，一个名为 `ENABLE_PROMPT_CACHING_1H` 的关键设置在 Claude Code 的发布说明中被提及但未出现在官方文档中。有人吐槽："天，越来越难追踪这些默认关闭的 MAKE_IT_ACTUALLY_WORK 设置了。"

**4. 公平比较的困难**：一位评论者提出模型在不同时间可能表现不同（即使版本号不变），因此"不同时间对同一模型的测试"才是"公平"的定义——这使得成本比较更加复杂。

## 行业影响与未来展望

这一事件凸显了 AI 产品定价的不透明性问题。用户为"同一个产品"付费，但底层 tokenizer 的变更实质性地改变了成本结构。更深层的问题是：当 AI 模型成为开发者的核心生产力工具后，供应商拥有了类似于操作系统级别的定价权。Max 5x 计划（约 $100/月）的用户发现自己的"配额"在新版本后实际缩水近一半，这种体验非常糟糕。

对于重度 Claude Code 用户，短期应对策略包括：使用 `ENABLE_PROMPT_CACHING_1H`、频繁 compact 会话、在不需要深度推理的任务上降低 effort 或切换到 Codex 等替代品。长期来看，这加速了开发者对多模型策略的需求。
