# Smol Machines：亚秒冷启动的便携虚拟机，Docker 的潜在替代者

**440 points | 134 comments**

## 事件背景

前 AWS 容器团队工程师 @binsquare 发布了 SmolVM——一个 CLI 工具，旨在用真正的硬件隔离 VM 替代 Docker 容器，同时保持容器级别的使用体验和亚秒级冷启动。项目开源（Apache-2.0），在 Show HN 中获得了极高关注。

## 核心观点/产品机制

SmolVM 的核心理念是"容器的便利性 + VM 的隔离性"：

- **亚秒冷启动**：< 200ms 启动一个完整的 Linux 虚拟机，对比 QEMU 的 15-30 秒
- **真正的硬件隔离**：每个工作负载运行在独立内核上，通过 macOS Hypervisor.framework 或 Linux KVM 实现
- **OCI 兼容**：直接使用 Docker Hub 上的镜像，无需 Docker 守护进程
- **可打包为便携二进制**：`.smolmachine` 格式可在任何支持平台运行，零依赖
- **弹性内存**：通过 virtio balloon 自动回收未使用内存
- **安全沙箱**：网络默认关闭，支持 `--allow-host` 白名单，SSH agent 转发不暴露私钥

底层基于 libkrun VMM 和自定义内核 libkrunfw。支持 macOS Apple Silicon / Intel 和 Linux x86_64 / aarch64。

作者明确定位：他在 AWS 工作期间意识到"容器是一个不必要的层，拖慢了速度"，而 Firecracker "是为 AWS 的组织结构和使用场景设计的"。SmolVM 是两者最佳部分的混合。

## 社区热议与争议点

**1. 实时迁移（Live Migration）是缺失的杀手功能**：一位资深用户强烈建议支持实时迁移——"这是所有类似系统总被忽略的功能。世界上有大量非云原生工作负载，因为云原生化成本太高。"他举例：遗留数据库 VM 需要迁移做维护、带状态的 Web 应用、长时间运行的训练任务。另一位有实现经验的工程师表示"实时迁移是我做过的最难的事"，但预测"12 个月内大部分开源沙箱实现都会通过 hypervisor 升级获得实时迁移"。

**2. 对 AI 沙箱场景的兴趣**：有开发者表示一直在为 AI 沙箱方案调研类似技术，但微型 VM 通常无法运行 Docker/Kubernetes。他问道 SmolVM 是否支持运行 k3s——这代表了一个重要的使用场景需求。

**3. AI 编码的争议**：有人问作者 SmolVM 有多少是用 AI 写的，得到回答"约 50%"。这引发了一场关于 AI 辅助编码可信度的小辩论。

## 行业影响与未来展望

SmolVM 填补了容器和传统 VM 之间的空白——对于需要强隔离但无法忍受 VM 启动延迟的场景（AI 代理沙箱、CI/CD、多租户计算），这是一个引人注目的方案。随着 AI 编码代理越来越多地执行不可信代码，"默认隔离"的理念正变得至关重要。SmolVM 的 Smolfile 声明式配置和 OCI 兼容性降低了采用门槛，但 GPU 支持仍在开发中，这是 AI 工作负载的关键短板。
