# 纯文本已经存在几十年，而且将继续存在

## 事件背景

Aresluna 博客发表了一篇关于"纯文本/ASCII"绘图和 UI 设计工具的文章，介绍了 Mockdown、Wiretext、Monodraw 等当代工具。这些工具让用户用等宽字符创建图表和界面原型，延续了 1970-80 年代 TUI（文本用户界面）的传统，但融入了现代特性：Web 访问、鼠标/触控板支持、高性能渲染。文章认为，纯文本的力量不仅在于文件格式的可移植性，更在于"文本编辑作为接口"的强大表达能力——而这种约束实践在 AI 时代将变得更加重要。

## 核心观点

文章提出了一个深刻洞见：随着计算机变得越来越强大，自我约束变得越来越重要——不仅是为了简化工作（让事情对你更容易），也是为了抵抗自动化的诱惑（让事情保持必要的难度）。纯文本作为一种极致的约束形式，反而释放了惊人的长寿性和可组合性。从源代码注释中的 ASCII 图表到 gen AI 的入口，纯文本在现代技术栈中的角色正在扩展而非萎缩。

## 社区热议

1. 最热门的回复来自一位从 QuickBooks 转向纯文本记账的用户："I just switched from QuickBooks to Beancount+Fava for my sole proprietorship, and couldn't be happier. I've added a text-based simple invoice system, a text-based vehicle mileage tracker... with git + RFC3161 attestation of commits, I can prove I made additions when I said I made them." 他甚至用 freetsa.org 和 OpenSSL 对 git commit 做了时间戳认证。

2. 关于 TUI 的黄金时代，有用户回忆 90 年代早期 DOS 时代："you had some very good TUIs in the early 1990's for DOS apps... QBASIC and EDIT.COM for example." 另一位补充道 "I always liked Borland's code editor from that era. The one that you used in Turbo-C, Turbo-Pascal, etc."

3. 整个技术栈向文本回归的趋势引发共鸣。有用户分享："I run ~15 personal tools and every one that survived past the first month stores data as JSON/markdown in git repos. Text in git gives you versioning, sync, grep, and you can hand the whole thing to an LLM with zero serialization."

4. TUI vs GUI 的辩论也很精彩。面对"为什么不用 4K 显示器的全部像素"的质疑，有人回应："Constraint breeds consistency and consistency breeds coherence... Take 1,000 random TUI designers and 1,000 random GUI designers... the TUI designers will be more tightly clustered together."

## 行业影响

纯文本工具的复兴反映了开发者社区对可控性和可预测性的渴望。在 AI 驱动的自动化浪潮中，纯文本作为"LLM 原生"的数据格式，其地位反而得到了强化——Markdown、JSON、Git 构成的工具链天然适合 AI 协作。同时，Beancount/Fava 等纯文本会计工具的成功，暗示着专业软件向"文件优先"（file-over-app）范式转移的可能性。
