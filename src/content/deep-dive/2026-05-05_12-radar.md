# Radar — The missing open-source Kubernetes UI

## 1. 事件背景
Kubernetes 已经是云原生事实标准，但「内置 dashboard」体验长期落后：Kubernetes Dashboard 简陋，Lens 闭源后社区质疑增多，K9s 是 TUI 不适合团队协作，Headlamp 还在成长期。OpenLens、Aptakube 也在抢占这一空白。Radar 在 Product Hunt 收获 380 票（weekly），自我定位是「missing OSS K8s UI」，瞄准 Lens 流走的开发者群体。

## 2. 核心架构与技术细节
Radar 的能力清单覆盖了运维 + 开发双角色：
- **实时拓扑**：service mesh 风格的资源依赖图。
- **资源/事件/Helm/GitOps** 一体视图（融合 Argo CD/Flux 概念）。
- **Live traffic flows**：基于 eBPF 或 sidecar 的实时调用链。
- **安全 & 最佳实践检查**：内嵌 kube-bench/kube-linter 风格规则。
- **Image filesystem inspection**：对镜像层做深度检查，支持 Trivy 风格漏洞扫描。
- **MCP for AI agents**：把上述能力暴露给 Claude/Cursor/OpenClaw，AI 可直接 query 集群状态、apply manifest。
部署形态非常务实：单二进制本地跑（开发者模式）或 in-cluster + RBAC + OIDC（团队模式），无需账号、无 SaaS 强制。

## 3. 竞品对比与生态站位
- vs **Lens**：Radar 完全开源 + MCP 原生，吸引对 Lens 商业化转向不满的用户。
- vs **K9s**：Radar 提供 GUI + 拓扑可视化，但保留 CLI 友好的快捷键。
- vs **Headlamp/Komodor**：Radar 把「AI Agent 集成」做成一等公民，时机踩中。

## 4. 开发者反馈与局限性
正方：单二进制部署 + 不要求 SaaS 账号，深得 SRE 圈青睐；MCP 接入让「让 Claude 帮我看下 prod 集群」成为现实工作流。
反方/局限：(1) 实时 traffic flows 取决于集群是否预装 Cilium/Istio，开箱即用程度待验证；(2) 安全检查规则集与企业策略对接需要更多 vendor extensibility；(3) 企业版（多集群 / 多租户审批）是否会闭源是社区担忧。
