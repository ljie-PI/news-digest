# HN 深度分析：Dav2d —— VideoLAN 的下一代 AV2 解码器

> 来源：Hacker News | 调研日期：2026-05-03

---

## 1) 事件背景

2025 年 10 月初，VideoLAN 基金会（VLC 播放器背后的开源组织）在其 GitLab 实例上悄然发布了 **dav2d** —— 下一代开放视频标准 **AV2** 的首个开源解码器实现。该项目延续了其前辈 dav1d（AV1 解码器，已被 Firefox、Chromium、VLC 等主流软件采用）的命名传统，在 Hacker News 上迅速获得 277 分、95 条评论的关注度，引发了技术社区对视频编解码未来的热烈讨论。

AV2 由 **AOMedia（Alliance for Open Media，开放媒体联盟）** 主导制定，是在 AV1 基础上演进的新一代视频编码规范，目标是通过更优的压缩算法，在同等画质下显著降低码率，从而优化流媒体、广播和实时视频会议的体验。而 dav2d 的目标则非常明确：**成为全平台最快的 AV2 解码器，同时保持小巧、可移植和极快的性能**。

---

## 2) 核心观点 / 产品机制

从项目描述来看，dav2d 延续了 dav1d 的工程哲学——**以极致性能为核心**。社区资深开发者 tux3 指出，dav1d 和 dav2d 的代码库实际上**主要由汇编（ASM）编写**，C 语言仅作为胶水代码或尚未优化部分的 fallback。由于 dav2d 尚处于早期阶段，其 C 代码比例相对更高，但汇编仍然是主导语言。

这一技术选择的逻辑十分清晰：视频解码是计算密集型的实时任务，每一帧的解码延迟都直接影响播放体验。在热点路径上使用手写 SIMD/汇编指令，是压榨硬件性能、实现跨平台一致低延迟的唯一可行路径。VideoLAN 在 dav1d 上已经验证了这一方法论的成功——dav1d 不仅在速度上击败了 Google 和 Mozilla 早期的 AV1 解码器，还成为了事实上的行业标准实现。

dav2d 的发布标志着 AV2 生态系统迈出了**从规范到可运行代码**的关键一步。不过，正如社区用户 Telaneo 所提醒的，参照 SVT-AV1 的历史经验，**成熟的编码器（encoder）通常比解码器来得慢得多**，距离 AV2 进入大规模商业应用还有相当长的一段路要走。

## 3) 社区热议与争议点

### 争议一：C/汇编 vs 内存安全语言——编解码器的"安全原罪"之争

这是评论区最激烈的分歧，正反双方都有强有力的论据。

**正方（主张迁移至内存安全语言）：**

用户 **dcsommer** 旗帜鲜明地表态：*"We must not continue to develop media codecs in memory unsafe languages... choosing default-unsafe for this type of software is close to professional negligence."*（我们不应继续使用内存不安全的语言开发媒体编解码器……为这类软件默认选择不安全语言接近职业疏忽。）这一观点代表了近年来系统软件领域对内存安全的高度关注，尤其是在浏览器和媒体播放器这类广泛部署的软件中，缓冲区溢出和 Use-after-free 漏洞的代价极为高昂。

**反方（捍卫 C/汇编的工程合理性）：**

用户 **fguerraz** 则给出了系统性的反驳：*"Cryptography and video codecs are notable exceptions, they put a lot of effort to making the code provably memory safe: no recursion, limited use of stack variables, no dynamic allocations, etc. As a result, memory safe languages bring nothing but trouble by making it non deterministic, that's especially true for crypto where compiler 'optimisations' guarantee you side channels attacks."*（密码学和视频编解码器是显著例外，它们投入了大量精力使代码可证明地内存安全：无递归、限制栈变量使用、无动态分配等。因此内存安全语言只会带来麻烦，使其非确定性——这在密码学中尤其严重，编译器"优化"几乎等于引入侧信道攻击。）

用户 **fishgoesblub** 则以事实说话：*"Of the 3 software AV1 encoders, the only one that is fully dead is the Rust encoder (rav1e). If people truly wanted memory safe encoders/decoders, they would fund and develop them."*（在 3 个软件 AV1 编码器中，唯一完全停滞的是 Rust 编码器 rav1e。如果人们真的想要内存安全的编码器/解码器，他们会出资开发它们。）

**折中立场：**

用户 **kllrnohj** 提供了一个务实的分层方案：*"For the codec itself... a memory safe language doesn't change much. However for the container/extractor... those should absolutely be in a memory safe language... As a practical example see CrabbyAVIF. All the parser code is rust, but it delegates to dav1d for the actual codec portion."*（编解码器核心本身性能敏感，内存安全语言改变不了什么。但容器/提取器绝对应该用内存安全语言……实际例子：CrabbyAVIF 的解析器用 Rust，但编码部分委托给 dav1d。）

这一争议还在继续发酵，后续跟帖中 **simonask** 质问 fguerraz 所说的"非确定性"定义不清，而 **vlovich123** 则反驳 rav1e 并非"完全死亡"，仍在活跃开发中。

### 争议二：AV2 相比 AV1 的实际增益与商业化前景

用户 **Telaneo** 表达了对 AV2 画质和压缩效率的期待，但也提醒：*"It's a shame it'll take a while before we'll have a decent encoder (it took an annoyingly long time until SVT-AV1 was usable)."*（遗憾的是，距离有一个靠谱的编码器还需要很长时间——SVT-AV1 花了很久才变得可用。）用户 **risho** 也追问：*"is there any understanding of how big of an improvment av2 will be over av1?"*（AV2 相比 AV1 有多大改进，现在有定论了吗？）

这反映了编解码器领域的一个核心矛盾：**标准制定和开源解码器实现可以很快，但高质量、高效率的编码器（encoder）开发才是决定一项技术能否落地的真正瓶颈**。编码器需要在压缩率、速度、质量之间做复杂的率失真优化，工程难度远高于解码器。

## 4) 行业影响与未来展望

dav2d 的发布具有三重行业意义：

**首先，它巩固了开源阵营在视频标准竞争中的地位。** 与需要支付专利费的 HEVC/H.266（VVC）不同，AV1/AV2 走的是免版税路线。VideoLAN 提供的开源、高性能解码器是这一战略落地的关键基础设施。dav1d 的成功已经证明，开源实现完全可以在性能上击败商业竞争对手。

**其次，它再次将"内存安全 vs 性能优先"的辩论推到了系统软件的最前线。** 随着 Rust 在 Linux 内核、Chrome 等关键项目中的渗透，媒体编解码器作为最后几块"C/汇编堡垒"之一，是否会被改写，将是一个值得长期观察的风向标。当前的折中方案（Rust 解析器 + C/ASM 编解码核心）可能是未来几年的主流模式。

**最后，AV2 的成熟仍需时间。** 虽然 dav2d 为解码端铺平了道路，但生态的真正爆发依赖于成熟的编码器（如 SVT-AV2 或 rav1e 的 AV2 版本）、硬件加速支持（GPU/专用芯片），以及内容提供商的采纳。参照 AV1 的采纳曲线，从解码器可用到 YouTube/Netflix 全面部署，大约需要 3-5 年。
