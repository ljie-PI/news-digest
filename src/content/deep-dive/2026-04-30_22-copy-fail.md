## 事件背景
copy.fail 是一个一页式的趣味/讽刺站点，在 2026-04-29 登上 HN 头版。站点内容主体围绕剪贴板（clipboard）的种种"失败"案例：从浏览器 Permission API 拒绝、Wayland 与 X11 不兼容、跨设备同步丢失格式、到 LLM 工具粘贴 prompt 时丢失代码块等等。作者用极简排版、动图和真实截图把"复制粘贴"这件每天都做的事的脆弱性做成了 meme 级展览。

## 核心观点
站点核心论点是：现代操作系统和浏览器层层加固之后，"复制粘贴"反而成了开发者最不可信任的基本功能。文章把问题分为五类：(1) 安全 API（Permissions/Focus 检查）误杀；(2) 富文本格式漂移（Markdown→Word→Email 多次往返失真）；(3) 移动端剪贴板 1 秒过期、跨设备同步失败；(4) 终端剪贴板（OSC 52）支持破碎；(5) AI 工具的"代码块吞噬"——把 ```python 等 fenced block 当成正文丢掉缩进。每节配可复现 demo。

## 社区热议
HN 评论区大致两派。一派把它当日常痛苦的宣泄："终于有人讲了"，并补充各自的灾难——VS Code 复制带颜色到 Slack 一锅端、macOS 通用剪贴板偶尔黑洞文件；另一派认为这是 Web 平台权衡的必然——为了挡住跨域剪贴板劫持只能引入复杂权限。也有人指出移动端 OS 厂商的"安全剪贴板提示"（Android 13+）其实是巨大的 UX 倒退。少数评论从可访问性角度提醒：屏幕阅读器和复制粘贴的兼容性更糟。

## 行业影响
copy.fail 不会立刻推动标准更新，但它把"剪贴板"重新放回到平台与浏览器厂商的视野里。短期内可见的趋势：W3C Async Clipboard 的兼容性修复、Wayland 1.5+ 的 wl-clipboard 一致性、以及编辑器内"AI 粘贴模式"（保留 markdown）的产品化。
